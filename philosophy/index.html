Philosophy

The config.js standard

We design the skins. You own the device logic. config.js is the handshake between them — one small file that makes any Gnoke skin work with any hardware that speaks JSON.

Think of config.js the way you think of Firebase's SDK initialisation — one standard connection file that any app can import to talk to the backend. You don't rewrite the app every time you change the backend. You just update the config.

Same idea here. You don't redesign the UI when you swap devices. You update config.js. The skin just works.

The standard file

Everything lives in config.js

One file. Declare your device address, the data keys it broadcasts, and the commands it accepts. That's the entire integration surface.

config.js — the only file you ever touch Edit once. Works everywhere. 
// 1. Where is your device?
export const device = {
  host:     "ws://192.168.1.42",   // WebSocket or HTTP
  protocol: "ws",               // "ws" | "http"
  pollMs:   500                // polling interval if HTTP
};

// 2. What data does it broadcast?
export const state = {
  duty:        { label: "Duty cycle", unit: "%",  min: 0, max: 100 },
  frequency:   { label: "Frequency",  unit: "Hz" },
  temperature: { label: "Temp",        unit: "°C" }
};

// 3. What commands does it accept?
export const commands = {
  setDuty:    { label: "Set duty",   param: "value" },
  loadPreset: { label: "Load preset", param: "value" }
};

// 4. Any preset shortcuts?
export const presets = [
  { id: "lathe", label: "Lathe", duty: 45 },
  { id: "pump",  label: "Pump",  duty: 80 }
];
The contract

Your device just needs to align

Any device that sends and receives these two JSON shapes can use any Gnoke skin. The backend is yours — we only care about the interface.

Device → Skin  ·  state broadcast
{
  "duty":        75,
  "frequency":  1200,
  "temperature": 34
}
Skin → Device  ·  command dispatch
{
  "action": "setDuty",
  "value":  80
}

// or
{
  "action": "loadPreset",
  "value":  "lathe"
}

That's it. Two JSON shapes. The device backend is entirely your concern — we make no assumptions about how it's implemented. ESP32, Pi, cloud API, Node server — any of them work as long as they speak this contract.

Principles

Six rules we don't break

00

The skin is yours — fully

All skins are released under the MIT licence — download, modify, brand, and ship in your own product. No fees, no attribution required. The point is low entry: grab a working UI and make it your own.

01

config.js is the only integration point

The skin never knows about your hardware. It only reads config.js. Swap the device, update the config — the skin is untouched.

02

One file, zero dependencies

A skin ships as a single index.html. No build step. No npm. No internet required. Copy it anywhere and open it.

03

Skins are interchangeable

Industrial touchscreen today, mobile browser tomorrow. Same config.js — completely different skin. No rewiring needed.

04

The UI doesn't live on the device

Phone, tablet, desktop browser, touchscreen — the skin runs anywhere with a browser. Your device just handles logic and JSON.

05

Offline-first always

Last-known state is cached locally. Disconnect the device — the skin still shows the last reading instantly.

06

Technician-readable in 10 minutes

If someone can't understand what a skin is doing in under 10 minutes, it's too complex. That's a bug we fix.
